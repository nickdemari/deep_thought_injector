    1. Add an optional `Set<_ServiceIdentifier>? resolutionStack` parameter to BOTH `locate<T>()` and `locateAsync<T>()`. Since `_ServiceIdentifier` is private to this file, external callers cannot construct this parameter -- it's effectively invisible to the public API. The default is `null` (callers don't pass it; first call initializes it internally).

    2. At the top of `locate<T>()`, before any factory invocation:
       ```dart
       final key = _ServiceIdentifier(T, name);
       final stack = resolutionStack ?? <_ServiceIdentifier>{};
       if (stack.contains(key)) {
         final chain = [...stack, key]
             .map((id) => id.name != null ? '${id.type}(${id.name})' : '${id.type}')
             .join(' -> ');
         throw VogonPoetryException('Circular dependency detected: $chain');
       }
       stack.add(key);
       ```
       Note: Use NON-const `VogonPoetryException(...)` here since the message is dynamic. Do not change existing `const` throws elsewhere.

    3. Dart's default `Set` is `LinkedHashSet` which preserves insertion order -- this gives us both O(1) lookup and ordered chain display from one data structure. Add a comment documenting this reliance: `// Dart's default Set (LinkedHashSet) preserves insertion order for chain display.`

    4. When `locate` delegates to parent scope, forward the resolution stack:
       ```dart
       if (serviceFactory == null && parent != null) {
         return parent!.locate<T>(name: name, resolutionStack: stack);
       }
       ```

    5. The factory invocation `serviceFactory.syncFactory!()` does NOT directly accept the stack -- the stack is threaded through because the factory closure itself calls `scope.locate<T>()` which re-enters this method. The stack is per-resolution-chain because it's passed as a parameter, not stored as instance state. However, since the factory closure captures `scope` and calls `scope.locate()` WITHOUT the `resolutionStack` parameter, we need a different approach for factory-triggered re-entry.

       **The correct approach:** Use a per-scope instance field `Set<_ServiceIdentifier>? _activeResolutionStack` that is set before invoking the factory and cleared after. This works because Dart is single-threaded -- no concurrent mutation within a single `locate` call chain. The field is:
       - Set to the current stack before calling `syncFactory!()`
       - Read at the top of `locate()` if `resolutionStack` parameter is null (falling back to the instance field)
       - Cleared after factory returns (in a finally block)

       Wait -- this won't work for parent scopes (different instance). Better approach: Thread the stack through the parameter AND use an instance field as the "current resolution context" that factory closures can pick up:

       **Simplest correct approach:** Add a `Set<_ServiceIdentifier>? _currentResolutionStack` field to `SubEthaScope`. At the top of `locate()`:
       ```dart
       final stack = resolutionStack ?? _currentResolutionStack ?? <_ServiceIdentifier>{};
       ```
       Before invoking a factory, set `_currentResolutionStack = stack` so that any re-entrant `locate()` call from within the factory picks it up. After factory returns, restore the previous value. When delegating to parent, pass the stack explicitly via the parameter.

       Implementation:
       ```dart
       Set<_ServiceIdentifier>? _currentResolutionStack;

       T locate<T>({String? name, Set<_ServiceIdentifier>? resolutionStack}) {
         final key = _ServiceIdentifier(T, name);
         final stack = resolutionStack ?? _currentResolutionStack ?? <_ServiceIdentifier>{};

         if (stack.contains(key)) {
           final chain = [...stack, key]
               .map((id) => id.name != null ? '${id.type}(${id.name})' : '${id.type}')
               .join(' -> ');
           throw VogonPoetryException('Circular dependency detected: $chain');
         }
         stack.add(key);

         // ... lookup serviceFactory ...
         // Parent delegation:
         if (serviceFactory == null && parent != null) {
           return parent!.locate<T>(name: name, resolutionStack: stack);
         }

         // Before invoking factory:
         final previousStack = _currentResolutionStack;
         _currentResolutionStack = stack;
         try {
           // ... factory invocation (singleton or transient) ...
         } finally {
           _currentResolutionStack = previousStack;
         }
       }
       ```

    6. Apply the SAME pattern to `locateAsync<T>()`:
       - Same `resolutionStack` parameter
       - Same `_currentResolutionStack` fallback
       - Same cycle check at top
       - Forward stack to parent: `parent!.locateAsync<T>(name: name, resolutionStack: stack)`
       - Set/restore `_currentResolutionStack` around factory invocation (use try/finally)
       - IMPORTANT: The `_currentResolutionStack` must be set before the `await` in the factory call. Since `locateAsync` is async, and Dart yields at `await`, ensure the stack is set synchronously before the factory `await`.

    7. Do NOT modify the Completer logic from Plan 01 -- the circular dependency check happens BEFORE the Completer check (if we're in a cycle, we should throw immediately, not wait on a Completer that will never complete).
